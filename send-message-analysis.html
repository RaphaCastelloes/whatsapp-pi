Análise da feature send message

Projeto: whatsapp-pi · foco: fluxo de envio de mensagens, do UI/Tool até Baileys.

Escopo analisado

  • src/services/message.sender.ts
  • src/services/whatsapp.service.ts
  • src/ui/menu.handler.ts
  • src/ui/message-reply.view.ts
  • whatsapp-pi.ts (tool send_wa_message e reply automático)
  • testes: tests/unit/message.sender.test.ts, tests/unit/message-reply.view.test.ts, tests/unit/menu.handler.test.ts, tests/unit/send-wa-message.tool.test.ts

Como fluxo funciona

1) Tool LLM: send_wa_message

Registrada em whatsapp-pi.ts. Ela valida status conectado, loga payload, chama whatsappService.sendMessage() e grava no recents quando sucesso.

2) UI menu e reply

menu.handler.ts e message-reply.view.ts usam sendMenuMessage(), com presença composing/paused e persistência em RecentsService.

3) Camada de transporte

WhatsAppService decide entre dois caminhos:

  • sendMessage() → delega para MessageSender, com retry e espera por conexão.
  • sendMenuMessage() → envia direto no socket, sem retry/backoff.

Pontos fortes

  • Boa separação de responsabilidade: UI, serviço e transporte estão bem divididos.
  • Consistência visual: presença de digitação antes/depois do envio.
  • Persistência de histórico: mensagem enviada vai para RecentsService.
  • Cobertura de teste boa: há testes para sucesso, falha, desconexão e roteamento de UI/tool.

Problemas / riscos

  • Duplicação de lógica: sendMessage() e sendMenuMessage() repetem presença + envio + tratamento de erro.
  • Marca π espalhada: MessageSender adiciona sufixo, o menu às vezes adiciona e às vezes não. Regra fica implícita.
  • Retry inconsistente: só caminho do MessageSender tem retry/backoff. UI usa envio direto.
  • Wait por conexão em polling: waitIfOffline() fica checando a cada 1s. Funciona, mas é bruto.
  • Falhas silenciosas: sendPresence() e markRead() engolem erro fora do verbose.

Leitura técnica

Hoje feature está funcional e coerente, mas com dívida de manutenção em torno de duas políticas de envio: envio com retry/branded suffix e envio simples para UI. Isso pode gerar comportamento diferente para mesma ação de usuário dependendo do ponto de entrada. 

Entrada UI / Tool
  → WhatsAppService
    → MessageSender OU sendMenuMessage
      → socket.sendMessage
        → recents.recordMessage

Recomendações

  • Centralizar pipeline: criar helper único para montar presença, retry, envio e pós-processamento.
  • Tornar regra do π explícita: nomear a política em constante/função. Ex.: appendBrandSuffix().
  • Unificar error model: normalizar erro de timeout/socket/conexão para UI mostrar mensagem padrão.
  • Cobrir UI com retry mínimo: ao menos falhas transitórias no menu/reply deveriam ser reattemptadas.
  • Trocar polling por evento: se serviço expuser sinal de conexão, remove wait ativo.

Conclusão

Base boa. Código entrega feature sem bagunça estrutural. Maior problema é duplicação de fluxo de envio e política implícita de branding/retry. Se unificar isso, feature fica mais previsível e fácil de evoluir.
