Plano de Desenvolvimento — Menu Message Send Reliability

Feature branch: 024-short-name-fix · Objetivo: estabilizar envio de mensagens pelo menu e diferenciar respostas originadas pelo Agent das disparadas pelo menu.

1. Resumo executivo

Problema central: envio via menu usa caminho frágil, diferente do restante da aplicação. A solução proposta unifica o pipeline de envio, centraliza regras de formatação por origem e mantém histórico de mensagens enviadas.

Valor para o usuário

  • Envio pelo menu fica confiável.
  • Feedback de sucesso/erro fica claro.
  • Agent e menu ficam distinguíveis pelo marcador π.

Decisão de arquitetura

  • Um único caminho compartilhado de envio.
  • Regras de retry e erro centralizadas.
  • Política de formatação aplicada em um só lugar.

5. Pesquisa e decisões

DecisãoRacionalAlternativas rejeitadas
Usar um caminho compartilhado de envio Evita drift de comportamento e centraliza retry/erro Manter envio direto no menu; retry separado só para menu
Preservar contrato atual de resultado Reduz impacto em UI, tools e testes Criar formato novo de retorno
Centralizar regra de formatação por origem Remove inconsistência e preserva distinção Agent/menu Manter formatação por fluxo sem regra clara
Manter histórico de enviados Valor já esperado pelo usuário Adiar atualização de recents

6. Modelo de dados

OutgoingMessage

Campos: recipient, text, timestamp, status, messageId, source, attempts.

Regras: texto não pode ser vazio; sucesso deve gerar histórico.

SendRule

Campos: formatação, retry, erro, presença.

Regras: mesmo conjunto para todos os entry points.

ConversationHistoryEntry

Campos: messageId, senderNumber, text, direction, timestamp.

Relação: criado somente após envio bem-sucedido.

7. Estratégia de entrega

MVP: corrigir o menu primeiro (US1).

Sequência: Foundation → menu confiável → consistência entre entry points → validação clara → polish.

12. Riscos principais

  • Duplicação de lógica: se menu continuar com rota própria, a inconsistência volta.
  • Política de π por origem: precisa ficar explícita para evitar surpresa e preservar distinção Agent/menu.
  • Retry inconsistente: UI e tool precisam alinhar comportamento.

13. Conclusão

Este plano ataca a causa raiz: envio fraco no menu e regras espalhadas. A abordagem mais segura é unificar o fluxo, manter contrato estável e aplicar validação/feedback consistentes, preservando a distinção de origem via π.
